# ADR-0001: База даних — SQLite vs PostgreSQL vs MongoDB

**Статус:** Ухвалено\
**Дата:** 12.04.2026\
**Автор:** Official Echo

## Контекст

Проєкт потребував постійного сховища для записів підписок, токенів підтвердження та тегів останніх релізів. Завдання вимагало зберігати всі дані застосунку в базі даних та автоматично застосовувати міграції структури БД під час запуску сервісу. Уся система також мала запускатися командою `docker compose up --build`. Розглядалися три варіанти: PostgreSQL, MongoDB та SQLite. На момент ухвалення рішення MongoDB вже активно використовувалася в паралельному проєкті, PostgreSQL був добре відомим варіантом, а SQLite застосовувався в попередньому проєкті приблизно рік тому.

## Рішення

**MongoDB — відхилено**

Вже використовується в іншому одночасному проєкті. Розподіл уваги між двома кодовими базами, що залежать від однієї системи, створював би зайве тертя. Дані підписок також є принципово реляційними — електронні адреси, репозиторії, токени, стан підтвердження — і не виграють від документної моделі.

**PostgreSQL — відхилено**

Надто важкий для масштабу цього проєкту. Вимоги до конфігурації, зокрема автентифікація, ролі, окремий серверний процес, пул з'єднань, несуть суттєві витрати на налаштування без жодної користі для моноліту з єдиним невеликим реляційним набором даних.

**SQLite + better-sqlite3 — обрано**

SQLite практично не потребує конфігурації та працює в процесі застосунку. База даних — це єдиний файл, що добре відповідає персистентності через Docker-том і шляху `:memory:`, який використовується в CI. `better-sqlite3` — найшвидший SQLite-драйвер для Node.js, що надає синхронний API та усуває асинхронну складність для простих CRUD-операцій. Під час запуску було увімкнено режим WAL для покращення паралелізму читання.

## Наслідки

- `docker compose up --build` працює без будь-якого сервісу бази даних поза контейнером застосунку
- Модель єдиного запису SQLite не є проблемою за такого масштабу; режим WAL справляється з наявним легким паралелізмом
- Міграція на клієнт-серверну базу даних буде необхідна, якщо сервіс масштабується до кількох екземплярів або вимагатиме масового паралельного запису

## Подальші міркування

Kysely розглядається як шар побудови запитів поверх `better-sqlite3` — типобезпечна конструкція SQL без ваги повноцінного ORM, зі збереженням синхронного драйвера. Якщо сервісу згодом знадобиться асинхронний ввід-вивід на рівні бази даних або перехід на розподілене розгортання, `better-sqlite3` можна замінити на асинхронний драйвер.
