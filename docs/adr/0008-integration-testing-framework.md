# ADR-0008: Інтеграційне тестування HTTP — Supertest vs інші

**Статус:** Ухвалено\
**Дата:** 17.05.2026\
**Автор:** Official Echo

## Контекст

Потрібен був спосіб тестувати маршрути, middleware, валідацію, сервісний шар та базу даних без запуску реального сервера на порту. Тести мають бути ізольованими, детермінованими та запускатися в тому самому процесі Vitest й працювати з фреймворком Express.

## Рішення

**node:http / fetch або axios fetch проти реального сервера — відхилено**

Запуск реального сервера на порту в тестах вносить конфлікти портів, залежність від порядку запуску, необхідність явного завершення сервера тощо. Використання `fetch` проти `localhost` також означає, що тест і сервер є окремими процесами, що ускладнює мокування модулів.

**axios-mock-adapter / nock — відхилено**

Ці інструменти перехоплюють вихідні HTTP-запити, а не вхідні. Вони не підходять для тестування власних ендпоінтів Express.

**pactum — відхилено**

pactum позиціонується як "REST API Testing Tool for all levels in a Test Pyramid", з підтримкою запитів, сценаріїв, схем та історії. Він працює з реальним HTTP‑сервером, а не з Express‑об’єктом у межах процесу, тому не відповідає вимогам "без запуску реального сервера на порту".

**Supertest — обрано**

Supertest приймає Express `app` безпосередньо і відкриває сервер без прив'язки до порту для кожного запиту. Це робить його об'єктивно найкращим варіантом для інтеграційного тестування Express: немає конфліктів портів, немає окремого процесу, повна підтримка `vi.mock()` для заглушення зовнішніх залежностей, і він чисто інтегрується з Vitest. API лаконічний та читабельний.

## Наслідки

- Інтеграційні тести живуть у `tests/integration/` та запускаються через `pnpm test:integration`
- Зовнішні залежності (GitHub API, nodemailer, Redis) мокуються через `vi.mock()` на рівні модуля
- База даних використовує in-memory SQLite (`:memory:`), що дає чисту ізоляцію без файлової системи
- Інтеграційні тести отримують сторінки через Supertest

## Подальші міркування

Supertest не підходить, якщо сервіс мігрує на Fastify з його іншою моделлю життєвого циклу, але навіть тоді існує `@fastify/inject` як пряма альтернатива з ідентичним концептом. В інакшому випадку у нього фактично немає суттєвих конкурентів.
