# ADR-0006: Тестовий фреймворк — Vitest vs Jest vs інші

**Статус:** Ухвалено\
**Дата:** 02.05.2026\
**Автор:** Official Echo

## Контекст

Проєкт використовує нативні синтаксис ESM. Потрібен був тестовий фреймворк, здатний запускати ці файли без трансформації або обхідних рішень. Юніт-тести бізнес-логіки були обов'язковою вимогою завдання; інтеграційні тести зазначалися як бонус. Фреймворк також мав забезпечувати надійну підтримку мокування для ізоляції таких сервісів, як GitHub API клієнт, шар бази даних та поштовий сповіщувач.

## Рішення

**Jest — відхилено**

Підтримка ESM у Jest є експериментальною та потребує або Babel-трансформації, або прапорця Node `--experimental-vm-modules`. Обидва підходи додають крихкість і конфігураційні витрати до проєкту, написаному повністю на JS. Невідповідність між внутрішнім CommonJS-ядром Jest та нативним ESM проєкту стала жорстким блокером.

**Mocha, AVA, node:test — відхилено**

Підтримують ESM, але не мають вбудованого покриття, мокування та ергономіки режиму перегляду, що постачаються з Vitest з коробки. Вбудований `node:test` Node.js ще розвивається та не має достатнього набору тверджень для тестового набору.

**Vitest — обрано**

Vitest створено спеціально для нативного ESM. Не потребує трансформації, нативно розуміє синтаксис `import` і забезпечує вбудоване покриття через `@vitest/coverage-v8`. Його API навмисно сумісний з Jest, тому синтаксис знайомий і переносний. Глобальні змінні (`describe`, `test`, `expect`, `vi`) широко відомі в екосистемі тестування JavaScript.

## Наслідки

- ESM-імпорти працюють без будь-якої Babel-конфігурації, прапорців Node або CJS-заглушок
- `vi.mock()` та `vi.resetModules()` чисто обробляють патерни динамічного імпорту, що використовуються в `auth.test.js`
- Покриття звітується через нативну інструментацію
- Глобальні змінні Vitest оголошено в `eslint.config.js`, тому лінтер розпізнає їх без явних імпортів у тестових файлах

## Подальші міркування

Міграція з Vitest не передбачається. Якщо проєкт перейде на TypeScript, Vitest обробляє `.ts`-файли нативно через esbuild. Скрипт `test:ci` вже використовує `--run --reporter dot` для чистого виводу в CI.
