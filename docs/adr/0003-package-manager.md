# ADR-0003: Менеджер пакетів — pnpm vs npm vs Bun vs Deno

**Статус:** Ухвалено\
**Дата:** 12.04.2026\
**Автор:** Official Echo

## Контекст

Потрібен був менеджер пакетів для встановлення залежностей, запуску скриптів та управління lockfile у локальній розробці, CI та Docker.

## Рішення

**Bun — відхилено**

Bun використовує рушій JavaScriptCore від Apple замість V8, що створює суттєві несумісності з нативними модулями Node.js — включно з `better-sqlite3`, який потребує компіляції проти заголовків Node. Більш суттєво, Bun — це комплексне середовище виконання, яке прагне одночасно контролювати тест-раннер, бандлер та менеджер пакетів. Це суперечить підходу проєкту — компонувати найкращі інструменти в своєму класі. Впровадження Bun означало б або прийняття всього його набору інструментів, або боротьбу з ним на кожному кроці.

**Deno — відхилено**

Модель дозволів Deno та історично відмінне розв'язання модулів додають тертя при роботі з npm-екосистемою. З прямого досвіду, Deno також показав нижчу продуктивність, ніж pnpm, у кількох операціях з залежностями. Його рівень сумісності з npm покращився, але він вносить зайвий шар абстракції для проєкту, що вже орієнтований на Node 20.

**npm — не обрано**

npm працює і був би безпечним варіантом за замовчуванням, проте економне з погляду дискового простору ставлення pnpm та 2-3 кратна перевага у швидкості дала про себе знати.

**pnpm — обрано**

pnpm забезпечує швидшу інсталяцію та суттєву економію диска через своє глобальне адресоване за вмістом сховище.

## Наслідки

- `pnpm install --frozen-lockfile` використовується як в CI, так і в Dockerfile, гарантуючи відтворювані збірки
- pnpm встановлюється через `corepack` у Dockerfile — не потрібно жодного додаткового шару інструментарію
- Використання диска між проєктами, що спільно використовують одне сховище pnpm, суттєво нижче, ніж з npm
- Всі скрипти запускаються через `pnpm exec` або `pnpm <script>`, забезпечуючи єдиний інтерфейс

## Подальші міркування

Якби проєкт був реструктурований як монорепозиторій (наприклад, з виокремленням gRPC-сервісу в окремий пакет), pnpm workspaces впорався б з цим нативно без зміни інструментів.
