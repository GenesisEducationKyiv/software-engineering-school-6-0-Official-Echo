# ADR-0010: Валідація вхідних даних — Zod 3 vs інші

**Статус:** Ухвалено\
**Дата:** 12.04.2026\
**Автор:** Official Echo

## Контекст

Сервіс приймає вхідні дані від користувачів через HTTP: адреси електронної пошти, назви репозиторіїв, токени. Усі вхідні дані потрібно валідувати та повертати конкретні, машиночитабельні коди помилок.

## Рішення

**Joi — відхилено**

Joi передує TypeScript і не має вбудованого виведення типів. Його API є ланцюжковим, але без перевірки типів. Для проєкту без TypeScript це менш критично, але Joi також більш багатослівний і має більший бандл, ніж Zod.

**Yup — відхилено**

Yup схожий на Joi. Менше активно розвивається, а його TypeScript-інтеграція є вторинною турботою, а не первинним дизайном.

**Valibot — не обрано**

Valibot є дуже легким конкурентом Zod із модульним дизайном для tree-shaking. Перспективний, але менш зрілий і з меншою екосистемою на момент вибору.

**Zod 4 — відхилено**

Zod 4 вийшов з суттєвими змінами, що погіршують зручність для цього проєкту. Зокрема, `required_error` видалено з `.string()` — розробник тепер має сам реалізовувати розрізнення між відсутнім і невалідним полем через `.preprocess()` або власну логіку. Аналогічно, ланцюжок `.string().min(1).email()` замінено на `.email()` без тонкого контролю над порядком та повідомленнями валідації — мінімальна довжина та перевірка формату більше не є окремими кроками з окремими повідомленнями. Ці зміни змушують реалізовувати вручну саме ту нюансну поведінку, яку Zod 3 надавав з коробки.

**Zod 3 — обрано**

Zod 3 надає повний контроль над повідомленнями про помилки через `required_error`, `invalid_type_error` та `message` на рівні кожного поля. Ланцюжок `.string().min(1, { message: ... }).email({ message: ... })` дозволяє розрізняти порожнє поле та невалідний формат із різними кодами помилок — саме це потрібно для машиночитабельних відповідей API. Схеми є декларативними і добре читаються.

## Наслідки

- Усі схеми валідації живуть у `src/validation/` і кидають `ValidationError` із відповідним кодом із `src/errors/constants/`
- Zod `3.x` закріплено в `package.json` для запобігання автоматичному оновленню до Zod 4
- Кожен ендпоінт валідує вхідні дані на межі, до будь-якої бізнес-логіки

## Подальші міркування

Міграція на Zod 4 можлива, якщо команда готова самостійно реалізовувати необхідні перевірки для даних замість фреймворку. Valibot залишається цікавою альтернативою при наступному аудиті залежностей, особливо якщо важливими стануть його підхід до ергономіки API, надійності та сучасних можливостей типізації, а також меншому результуючому розмірі.
