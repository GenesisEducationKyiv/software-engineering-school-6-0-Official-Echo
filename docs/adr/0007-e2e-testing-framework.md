# ADR-0007: E2E-тестування — Playwright vs Cypress vs інші

**Статус:** Ухвалено\
**Дата:** 17.05.2026\
**Автор:** Official Echo

## Контекст

Завдання явно вимагало E2E-тестів з Playwright для сторінки застосунку. Потрібен був інструмент браузерного тестування, здатний завантажувати реальний UI, взаємодіяти з формами та перехоплювати мережеві запити для заглушення API-відповідей.

## Рішення

**Selenium — відхилено**

Selenium — найстаріший варіант і вимагає окремого сервера WebDriver та ручного управління браузерними драйверами. Його архітектура додає суттєві витрати на налаштування порівняно з сучасними інструментами. Для нового проєкту немає жодних причин обирати його.

**Puppeteer — відхилено**

Puppeteer керує лише Chromium і підтримується командою Chrome. Не є повноцінним тестовим фреймворком — не має вбудованих тверджень, раннера, паралелізму чи звітування.

**Cypress — відхилено**

Cypress — сучасний, добре документований інструмент E2E з відмінним інтерактивним UI-раннером та чистим синтаксисом. Він більш орієнтований на розробницький досвід і має сильну екосистему. Проте завдання явно вказувало на Playwright.

**Playwright — обрано**

Playwright вимагався завданням. Технічно він підтримує Chromium, Firefox та WebKit в єдиному API, має вбудоване перехоплення мережі через `page.route()` та офіційний Docker-образ `mcr.microsoft.com/playwright`, що спрощує CI. `@playwright/test` поставляється з власним раннером, тому не потребує інтеграції з Vitest.

## Наслідки

- E2E-тести живуть у `tests/e2e/` та запускаються окремо від Vitest через `pnpm test:e2e`
- `playwright.config.js` обмежує `testDir` до `tests/e2e/`, щоб уникнути конфліктів з тестовими файлами Vitest
- Сервер запускається автоматично через опцію конфігурації `webServer`
- Браузери потрібно встановити один раз локально: `pnpm exec playwright install chromium`
- В Docker браузер встановлюється автоматично через `Dockerfile.e2e`

## Подальші міркування

У майбутньому можна перейти на Cypress через його інтерактивний раннер або кращий досвід налагодження, обидва інструменти підтримують перехоплення мережі та роботу з формами в схожому стилі. Структура тестів залишилася б практично ідентичною. У випадку, якщо буде ухвалено рішення йти тільки з Jest/Vitest можна розглядати Pupeteer для майбутнього E2E тестування.
