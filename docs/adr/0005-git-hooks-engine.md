# ADR-0005: Git-хуки — Lefthook vs Husky vs simple-git-hooks

**Статус:** Ухвалено\
**Дата:** 02.05.2026\
**Автор:** Official Echo

## Контекст

Потрібна була автоматизація pre-commit та pre-push для забезпечення лінтингу, форматування та аудиту залежностей перед тим, як код потрапляє до віддаленого репозиторію. Завдання вимагало CI-пайплайн GitHub Actions як додатковий результат, але локальне виконання хуків було окремим самостійно встановленим контролем якості.

## Рішення

**simple-git-hooks — відхилено**

`simple-git-hooks` — мінімальний пакет, що безпосередньо відображає імена Git-хуків на shell-команди у `package.json`. Він навмисно обмежений: без паралелізму, без фільтрації за glob, без виконання для окремих файлів, без автоматичного індексування. Для єдиної команди `lint && format` він би спрацював, але проєкт потребував паралельного виконання ESLint і Prettier лише для індексованих файлів, плюс окремого кроку аудиту pre-push. Вираження цього без окремого конфігураційного файлу вимагало б shell-операцій, складніших для читання та підтримки, ніж спеціалізований інструмент.

**Husky — відхилено**

Husky — найпоширеніший менеджер Git-хуків в екосистемі Node.js, але він накопичив складність конфігурації між мажорними версіями — з v4 до v7 підхід до налаштування кардинально змінювався. Він потребує хука lifecycle `prepare` у `package.json` та окремих файлів хуків для кожної події. Порівняно з єдиним декларативним конфігураційним файлом Lefthook, він не дає жодних переваг для цього випадку використання і вносить більше рухомих частин.

**Lefthook — обрано**

Lefthook написаний на Go і суттєво швидший за JavaScript-альтернативи. Єдиний `lefthook.yml` оголошує всі хуки з їхніми glob-патернами, командами та налаштуваннями паралелізму. Опція `parallel: true` у блоці pre-commit запускає ESLint і Prettier паралельно лише для індексованих файлів, мінімізуючи затримку під час коміту. `stage_fixed: true` автоматично переіндексовує автоматично виправлені файли, усуваючи необхідність ручного `git add` після виправлення лінтером. Хук pre-push запускає `pnpm audit --audit-level=high` один раз на пуш, тримаючи його поза швидким шляхом коміту.

## Наслідки

- Хуки оголошені в `lefthook.yml` та знаходяться під контролем версій
- Паралельне виконання перевірок лінтингу та форматування лише для індексованих файлів тримає час pre-commit мінімальним
- Автоматичне переіндексування виправлених файлів усуває поширену причину відключення хуків
- Бінарний файл Go не має залежності від Node.js під час виконання, тому хуки працюють навіть при частково зламаному середовищі Node
- Аудит pre-push виявляє вразливості високої серйозності до того, як вони потрапляють до віддаленого репозиторію, доповнюючи CI-пайплайн GitHub Actions

## Подальші міркування

Lefthook підтримує конфігурації хуків з віддалених джерел (завантаження визначень хуків з URL), що могло б бути корисним, якби цей проєкт став частиною більшої організації зі спільними стандартами лінтингу. Такої потреби наразі немає.
